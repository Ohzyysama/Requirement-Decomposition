# 需求代码定位测试集 —— 说明与评分口径

> 任务方向：**给定一个 commit（代码快照）+ 一条 AR 需求描述 → 定位这个 AR 关联了哪些代码**。
> 标准答案对齐**新 15-SR / 77-AR 方案**（见 `../requirements/SR-AR-mapping.md`）。

## 1. 数据集构造

- 分支：`eval/requirement-testset`（从 `main` 切出）。
- 在需求文档提交（15-SR/77-AR）之上，为每个 AR 各堆叠提交，分**三层**，共 **174 个用例**：

### 简单层 L01–L77（77）
- 每个提交向该 AR 的锚点文件追加**两段真实 ArkTS 代码**：① **AR 真实实现**（标准答案）；② 一段与该 AR 无关的**通用干扰函数**。

### 难例层 H01–H77（77）
- 每个提交向锚点文件追加 **1 段 AR 真实实现 + 5 段干扰**，且**目标实现位置被打乱**（不固定在首位）：
  - **3 段是其它 AR 的真实代码**（优先取同 SR 的相邻 AR，作为同域强干扰），标准答案中以 `source` 标注**来源 AR 编号**；
  - **2 段是通用无关代码**（`source: "generic"`）。
- 考查系统在「同一 commit/文件里混入其它 AR 的真实实现」时，能否只挑出目标 AR 的代码、不被同域相似代码带偏。

### 复杂层 X01–X20（20）
四类更复杂的场景（生成器 `_gen/commit-complex.mjs`）：

| 场景 | 用例 | 形态 | 考查点 |
|------|------|------|--------|
| `multi` 同文件多段实现 | X01–X06 | **2 段正例** + 3 段干扰（同文件） | 多段实现的"找全"能力（召回） |
| `xfile` 跨文件实现 | X07–X12 | **2 段正例分布在 2 个文件** + 4 段干扰跨文件分布 | 跨文件定位与找全 |
| `xsr` 跨 SR 同语义干扰 | X13–X16 | 1 正例 + 5 干扰（3 段为**其它 SR 同语义家族** AR 的真实代码，如歌曲/专辑/歌手/文件夹的 ABC 索引、搜索、排序、多选互为干扰） | 跨域同语义代码的区分能力 |
| `large` 大候选集 | X17–X20 | 1 正例 + **9 段干扰**（7 跨 AR + 2 通用），位置打乱 | 大候选集下的精确率 |

- 标准答案 `ground-truth.json` 为**样本模板结构**（见下），每个样本含 `commit`（真实 SHA）、AR 查询文本与全部候选片段（正例 + 干扰）的 **文件 + 符号 + 行号区间** 与 `label` 标注；层级由 `sample_id` 前缀区分（`L`/`H`/`X`）。
- 旧版标注结构（含 `tier`、干扰 `source` 来源 AR 等）保留在 `_gen/ground-truth.legacy.json`，是文档与样本的单一数据源。
- 改动均为追加式（只在文件末尾加新函数），不破坏既有逻辑；行号在该 commit 与后续快照中保持稳定。
- 生成器见 `_gen/`（snippets 为各 AR 代码规格，`commit.mjs` 建简单层，`commit-hard.mjs` 建难例层，`commit-complex.mjs` 建复杂层，`samples.mjs` 由 legacy 生成样本结构 `ground-truth.json`，`docs.mjs` 出文档），可复现。

## 2. 单条样本的形态（`ground-truth.json`）

| 字段 | 含义 |
|------|------|
| `sample_id` | 样本号 L01–L77 / H01–H77 / X01–X20 |
| `project` / `commit` | 项目名与该 AR 实现所在提交的真实 SHA（评测输入之一） |
| `query_ar_id` / `query_ar_title` / `query_ar_text` | AR 编号、标题与完整需求描述（评测输入之一） |
| `raw_positive_count` / `raw_candidate_count` | 正例数 / 候选总数 |
| `candidate_snippets[]` | 候选片段：`{candidate_id, file_path, symbol, start_line, end_line, content, label, original_index}` |
| `candidate_snippets[].label` | **1 = 正例**（该 AR 的实现，标准答案）；**0 = 干扰**（系统命中即误报） |

> 输入 = `commit + AR 需求文本`；期望输出 = 全部 `label=1` 候选的代码位置；`label=0` 是同一 commit 内的干扰，用于考查系统能否区分 AR 实现与无关代码。复杂层 `multi`/`xfile` 场景每样本有 **2 段正例**，须全部命中才算找全。

## 3. 评分口径

对每条用例，将系统输出的代码位置与标准答案 `implementation` 对照：

- **文件级准确率** = 定位到正确文件的用例数 / 77。
- **符号级准确率** = 定位到正确符号（函数/类名）的用例数 / 77。
- **行级 IoU** = 预测行区间 ∩ 标准行区间 / 预测行区间 ∪ 标准行区间，对 77 条取平均。
- **干扰命中率** = 错误地把干扰代码（`distractor` 的符号/行区间）判为该 AR 实现的用例数 / 77，理想为 0。
- **综合命中** = 符号正确 **且** 未命中干扰 的用例数 / 77。

建议同时报告文件级 / 符号级 / 行级三档，定位系统在「同文件多函数中精确到符号与行」上的能力，以及抗干扰能力。

## 4. 报告建议

- 总表：每用例一行，列出 `case / ar_id / commit / 标准答案 / 系统输出 / 文件✓ / 符号✓ / 行级IoU / 命中干扰?`（见 `score-template.md`）。
- 汇总：文件级、符号级准确率，行级平均 IoU，干扰命中率，综合命中率。
- 可按 SR 分组报告各域定位准确率，定位系统在列表/详情/设置等相似代码簇上的区分能力。
