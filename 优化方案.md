# 需求拆分评估与重拆优化方案

---

## 一、优化背景与问题诊断

### 1.1 当前流程概览

```
M1 拆分（整棵功能树）
    ↓
依赖分析
    ↓
生成子需求列表
    ↓
M2 一致性评估  ←── 问题一：全量扁平评估，不感知树结构
    │
    ├── 通过 → M2 可实现性评估  ←── 问题三：warning 也触发自动重拆
    │              │
    │              ├── 通过 → 集成 → 完成
    │              └── 不通过 → 自动细化（往往过度拆分）
    │
    └── 不通过 → 整棵树重新 M1 拆分  ←── 问题二：整树推倒重拆
                  ↓
              重新依赖分析 → 重新一致性评估 → 循环
```

### 1.2 三大核心问题

| 编号 | 问题 | 影响 |
|------|------|------|
| **问题一** | 一致性评估为**全量扁平评估**，LLM 不感知树层级关系，`per_child` 字段闲置 | 评估粗粒度，前端无法按节点展示具体问题 |
| **问题二** | 一致性失败后**整棵功能树推倒重拆**，无视 `affected_nodes`（已标注具体哪些节点有问题） | LLM Token 浪费严重，通过节点可能被改坏，耗时长 |
| **问题三** | 可实现性评估中，**warning 级规则（技术复杂性、工作量偏高等）也触发自动重拆**，且阈值偏严 | 常见复杂功能被过度拆分，树无限下钻 |

### 1.3 问题根源分析

**问题一的根因**：[m2_consistency_agent.py](app/services/agents/M2/agents/m2_consistency_agent.py) 将整个扁平 `function_list`（可能 20+ 节点、3 层嵌套）一次性丢给 LLM，且 `ConsistencyEvaluationResult` 中的 `per_child` 字段始终硬编码为空：

```python
per_child=[]  # TODO 暂时为空，可根据需要扩展
```

**问题二的根因**：[pipeline_runner.py](app/services/coordinator/pipeline_runner.py) 的一致性重试循环中，评估结果明明包含 `affected_nodes`，却直接调用 `invoke_m1_decomposer_and_dependency_no_gate` 把整棵树推倒重拆：

```python
# 旧代码 — 无视 affected_nodes，整棵树重拆
dd = await self.invoke_m1_decomposer_and_dependency_no_gate(
    context, on_agent_complete=on_agent_complete
)
```

**问题三的根因**：[agent_invoker.py](app/services/coordinator/agent_invoker.py) 的 `_extract_feasibility_failed_node_ids` 收集所有 `passed=False` 规则的 `affected_nodes`，不区分 `severity=error` 和 `severity=warning`，也不区分 `issue_type`。同时阈值偏严（AFP > 15、工作量 > 0.5 人月、EO/ILF 即判复杂）。

## 二、优化思路总览

优化分为三个互补维度，形成完整闭环：

```
┌──────────────────────────────────────────────────────────────────┐
│  维度一：一致性评估逐节点层级化（评估侧）                            │
│  · LLM Prompt 改为要求输出 per_child_findings                      │
│  · 用户消息改为按父节点分组的树结构展示                               │
│  · _build_per_child_hints() 融合规则引擎 + LLM 语义分析              │
│  · ConsistencyEvaluationResult.per_child 写入真实数据                │
└───────────────────────────┬──────────────────────────────────────┘
                            │
                            ▼ per_child 列表（精准定位问题节点）
┌──────────────────────────────────────────────────────────────────┐
│  维度二：一致性重试增量精准化（重拆侧）                               │
│  · 优先使用 per_child hints 逐节点重拆                              │
│  · 次选从 affected_nodes 提取失败节点（含向上归约去重）               │
│  · 复用 SubARRefiner._refine_one_sub_ar_node 已有能力               │
│  · 双层兜底：无具体节点时回退全树重拆（100% 向后兼容）                 │
└───────────────────────────┬──────────────────────────────────────┘
                            │
                            ▼ 解决"一致性评估→重拆"的过度消耗
┌──────────────────────────────────────────────────────────────────┐
│  维度三：可实现性评估规则合理化（规则侧）                              │
│  · 区分 error / warning 触发条件：warning 只入报告不触发重拆         │
│  · 放宽阈值（AFP 15→25, 工作量 0.5→1.0, 复杂度判定精细化）           │
│  · 修正配置文案不一致（默认值 3→1）                                  │
│  · 粒度问题区分 too_coarse / too_fine（过细不进入拆分流程）           │
└──────────────────────────────────────────────────────────────────┘
```

### 核心原则

> **评估侧**：对每个子功能节点独立评估，生成精准的逐节点问题报告
>
> **重拆侧**：只重拆真正需要拆的问题节点，通过节点保持不动
>
> **规则侧**：warning 只作报告建议，不驱动自动拆分；仅 error 级或明确"规模严重过大"才触发自动细化

```python
class PerChildSemanticFinding(BaseModel):
    """LLM 对单个子功能节点的语义评估结果"""
    node_id: str = Field(..., description="子功能节点 ID")
    node_title: str = Field(default="", description="子功能节点标题")
    coverage_ok: bool = Field(default=True, description="覆盖完整性")
    faithfulness_ok: bool = Field(default=True, description="拆分忠实性")
    scope_ok: bool = Field(default=True, description="范围约束")
    issue_description: Optional[str] = Field(default=None, description="问题描述")
    suggestion: Optional[str] = Field(default=None, description="改进建议")


class LLMSemanticAssessmentResult(BaseModel):
    # ... 原有 5 个字段不变 ...
    per_child_findings: List[PerChildSemanticFinding] = Field(
        default_factory=list,
        description="逐子节点语义评估结果",
    )
```

#### 3.3.2 用户消息改为树结构展示

新增 `_group_function_list_by_parent` 静态方法，将扁平列表转为按父节点分组：

```python
@staticmethod
def _group_function_list_by_parent(
    function_list: List[Dict[str, Any]]
) -> str:
    """将扁平功能列表按 parent_id 分组，生成层级化描述文本"""
    if not function_list:
        return "（无功能节点）"

    # 按 parent_id 分组
    by_parent: Dict[str, List[Dict[str, Any]]] = {}
    id_to_title: Dict[str, str] = {}
    for row in function_list:
        nid = str(row.get("id", "")).strip()
        title = str(row.get("title", row.get("name", ""))).strip()
        if nid:
            id_to_title[nid] = title
        pid = str(row.get("parent_id", "")).strip()
        by_parent.setdefault(pid, []).append(row)

    parts: List[str] = ["功能树结构（按父节点分组展示）："]
    for pid, children in sorted(by_parent.items()):
        if not pid:
            label = "▸ 根层节点（无父节点）"
        else:
            parent_title = id_to_title.get(pid, pid)
            label = (
                f"▸ 父节点 [{pid}]「{parent_title}」"
                f"的直接子节点（共 {len(children)} 个）"
            )
        parts.append(label)
        for child in children:
            cid = child.get("id", "")
            ctitle = child.get("title", child.get("name", ""))
            ctype = child.get("node_type", child.get("type", ""))
            desc = child.get("description", "")
            parts.append(f"    · [{cid}] {ctitle} (类型:{ctype}) — {desc}")
    return "\n".join(parts)
```

**改造前 vs 改造后**：

```
改造前（扁平列表 dump）:
  标准化原需求文本：...
  功能列表：[{id: F-1, ...}, {id: F-1.1, ...}, ...]

改造后（树结构展示）:
  标准化原需求文本：...
  功能树结构（按父节点分组展示）：
  ▸ 根层节点（无父节点）：
    · [F-1] 在线课程平台
  ▸ 父节点 F-1「在线课程平台」的直接子节点（共 3 个）：
    · [F-1.1] 用户管理 (类型:MODULE) — 学生注册登录...
    · [F-1.2] 课程管理 (类型:MODULE) — 浏览课程列表...
    · [F-1.3] 订单支付 (类型:MODULE) — 购买课程...
```

#### 3.3.3 核心聚合方法 `_build_per_child_hints`

融合**规则引擎 affected_nodes** + **LLM 语义 per_child_findings**，生成逐节点精炼提示：

```python
def _build_per_child_hints(
    self,
    function_list: List[Dict[str, Any]],
    rule_results: List[RuleCheckResult],
    llm_per_child_findings: Optional[List[PerChildSemanticFinding]],
) -> List[PerChildRefinementHint]:
    """
    融合规则引擎（affected_nodes）+ LLM 语义分析（per_child_findings），
    生成逐节点精炼提示。
    """
    # 数据源 A：规则引擎 → 按 node_id 聚合
    rule_problems: Dict[str, List[str]] = {}
    rule_suggestions: Dict[str, List[str]] = {}

    for rr in rule_results:
        if rr.passed:
            continue
        for nid in (rr.affected_nodes or []):
            sid = str(nid).strip()
            if not sid:
                continue
            if rr.description:
                rule_problems.setdefault(sid, []).append(rr.description)
            if rr.suggestion:
                rule_suggestions.setdefault(sid, []).append(rr.suggestion)

    # 数据源 B：LLM 语义分析 → 提取非通过的节点
    llm_problems: Dict[str, PerChildSemanticFinding] = {}
    for pf in (llm_per_child_findings or []):
        if not pf.coverage_ok or not pf.faithfulness_ok or not pf.scope_ok:
            llm_problems[pf.node_id] = pf

    # 聚合：取两类数据源的问题节点并集
    all_problem_ids = set(rule_problems.keys()) | set(llm_problems.keys())

    hints: List[PerChildRefinementHint] = []
    for nid in sorted(all_problem_ids):
        reasons: List[str] = []
        instructions: List[str] = []

        # 规则引擎描述
        for desc in rule_problems.get(nid, []):
            reasons.append(f"[规则检查] {desc}")
        for sug in rule_suggestions.get(nid, []):
            instructions.append(f"[规则建议] {sug}")

        # LLM 语义分析
        pf = llm_problems.get(nid)
        if pf:
            if pf.issue_description:
                reasons.append(f"[语义分析] {pf.issue_description}")
            if pf.suggestion:
                instructions.append(f"[改进建议] {pf.suggestion}")

        # 判断是否需要进一步拆分
        need_split = bool(
            any(rr.level == "ERROR" for rr in rule_results
                if nid in (rr.affected_nodes or []))
            or (pf and not all([pf.coverage_ok, pf.faithfulness_ok, pf.scope_ok]))
        )

        hints.append(PerChildRefinementHint(
            node_id=nid,
            need_further_split=need_split,
            split_reason="；".join(reasons) if reasons else "一致性检查未通过",
            split_instruction="；".join(instructions) if instructions else "",
        ))

    return hints
```

**降级行为**：LLM 未返回 `per_child_findings` 时仅使用规则引擎数据，确保功能不退化。

#### 3.3.4 `execute` 方法串联完整流程

### 4.2 新增辅助函数 `_extract_failed_node_ids`

从评估结果中提取未通过的 `affected_nodes`，去重并**向上归约**为最小覆盖父节点集合：

```python
@staticmethod
def _extract_failed_node_ids(
    eval_dict: Dict[str, Any],
    current_function_list: Optional[List[Dict[str, Any]]] = None,
) -> List[str]:
    """
    从一致性/可实现性评估结果中提取未通过检查的 affected_nodes，
    去重并向上归约为最小覆盖父节点集合。

    归约规则：若节点 A 和 A 的祖先节点都在失败集，只保留祖先（最靠近根的失败节点），
    避免对同属于一个父的多个子节点分别重拆导致冲突覆盖。
    """
    failed_ids: set = set()

    # 收集所有未通过的 affected_nodes
    def _collect(item: Any) -> None:
        if not isinstance(item, dict):
            return
        if item.get("passed") is True:   # 显式通过的跳过
            return
        for nid in item.get("affected_nodes") or []:
            s = str(nid).strip()
            if s:
                failed_ids.add(s)

    # 从 critical_issues、rule_results、warnings 三处收集
    for item in eval_dict.get("critical_issues") or []:
        _collect(item)
    for item in eval_dict.get("rule_results") or []:
        _collect(item)
    for item in eval_dict.get("warnings") or []:
        _collect(item)

    if not failed_ids:
        return []

    # 构建 id → row 索引，用于逆查树关系
    id_to_row: Dict[str, Dict[str, Any]] = {}
    for row in (current_function_list or []):
        if isinstance(row, dict):
            nid = str(row.get("id", "")).strip()
            if nid:
                id_to_row[nid] = row

    # 向上归约：若某节点及其祖先都在失败集，移除子孙，只保留最顶层祖先
    result = set(failed_ids)
    for nid in list(result):
        row = id_to_row.get(nid, {})
        pid = str(row.get("parent_id", "")).strip()
        current = pid
        while current:
            if current in result:
                result.discard(nid)     # 祖先也在失败集 → 移除子孙
                break
            parent_row = id_to_row.get(current, {})
            current = str(parent_row.get("parent_id", "")).strip()

    return sorted(result)
```

**向上归约示例**：

```
功能树结构：              一致性失败节点：
  F-1                     ├── SR-01     ← 失败
  ├── SR-01               ├── SR-01-01  ← 也失败了（SR-01 的子节点）
  │   ├── SR-01-01        └── SR-02     ← 失败
  │   └── SR-01-02
  └── SR-02

归约后：只保留 SR-01 和 SR-02
（SR-01-01 因为其父 SR-01 也在失败集，被移除；重拆 SR-01 时会自然覆盖 SR-01-01）
```

**改动后（新代码）**：

```python
inner_attempt += 1
remediation = (ev.get("remediation_instruction") or "").strip()

# ───── 提取 per_child 问题节点（维度一的产出）─────
per_child = ev.get("per_child") or []
problem_hints = [h for h in per_child if h.get("need_further_split")]

# 祖先去重：按子树大小降序排列，排除被祖先覆盖的后代
if problem_hints:
    covered_ids: Set[str] = set()
    top_hints: List[Dict[str, Any]] = []
    sorted_hints = sorted(
        problem_hints,
        key=lambda h: len(collect_subtree_ids_from_rows(
            h["node_id"], current_function_list
        )),
        reverse=True,  # 子树越大越靠前（祖先优先）
    )
    for hint in sorted_hints:
        nid = hint["node_id"]
        if nid in covered_ids:
            continue  # 已被祖先覆盖，跳过
        subtree = collect_subtree_ids_from_rows(nid, current_function_list)
        covered_ids.update(subtree)
        top_hints.append(hint)
    problem_hints = top_hints

# 同时从 evaluation 结果中提取 affected_nodes（兼容旧版无 per_child 的情况）
current_fl = context.artifacts.get("function_list") or []
failed_node_ids = self._extract_failed_node_ids(ev, current_fl)

context.retry_context = {
    "consistency_passed": False,
    "consistency_inner_attempt": inner_attempt,
    "last_consistency_score": ev.get("score"),
    "remediation_instruction": remediation,
    "failed_node_ids": failed_node_ids,   # 【新增】
    "per_child_hints": problem_hints,      # 【新增】
}
ncfg = dict(context.config or {})
if remediation:
    ncfg["split_retry_hints"] = remediation
context.config = ncfg

if inner_attempt > inner_max:
    context.add_quality_flag(
        "consistency_evaluator",
        "consistency_not_passed_after_inner_retries",
    )
    break

# ───── 三路分支：per_child → affected_nodes → 全树重拆（兜底）─────
if problem_hints:
    # 【优先】使用 per_child 精准重拆（维度一的完整产出）
    logger.info(
        f"[{context.conversation_id}] 一致性未通过，"
        f"per_child 问题节点: {[h['node_id'] for h in problem_hints]}，"
        f"精准重拆第 {inner_attempt}/{inner_max}"
    )
    context.tree_version += 1
    context.add_artifact("tree_version", context.tree_version, agent="coordinator")

    for hint in problem_hints:
        if context.stop_requested:
            break
        hint_data = {
            "need_further_split": True,
            "split_reason": hint.get("split_reason", "consistency_failed"),
            "split_instruction": hint.get("split_instruction", remediation or ""),
        }
        ok = await self.sub_ar_refiner._refine_one_sub_ar_node(
            context, merged, hint["node_id"], hint_data, on_agent_complete
        )
        if not ok:
            logger.warning(
                f"[{context.conversation_id}] 精准重拆节点 {hint['node_id']} 失败，跳过"
            )

elif failed_node_ids:
    # 【次选】使用 affected_nodes 增量重拆（兼容旧版无 per_child）
    logger.info(
        f"[{context.conversation_id}] 一致性未通过，"
        f"affected_nodes: {failed_node_ids}，"
        f"增量重拆第 {inner_attempt}/{inner_max}"
    )
    context.tree_version += 1
    context.add_artifact("tree_version", context.tree_version, agent="coordinator")

    for nid in failed_node_ids:
        if context.stop_requested:
            break
        hint_data = {
            "need_further_split": True,
            "split_reason": "consistency_failed",
            "split_instruction": remediation or "一致性检查未通过，请对该子需求进一步细化拆分",
        }
        ok = await self.sub_ar_refiner._refine_one_sub_ar_node(
            context, merged, nid, hint_data, on_agent_complete
        )
        if not ok:
            logger.warning(
                f"[{context.conversation_id}] 增量重拆节点 {nid} 失败，跳过"
            )

else:
    # 【兜底】无具体失败节点信息，回退全树重拆（100% 保留原逻辑）
    logger.info(
        f"[{context.conversation_id}] 一致性未通过（无具体失败节点），"
        f"回退全树重拆第 {inner_attempt}/{inner_max}"
    )
    context.tree_version += 1
    context.add_artifact("tree_version", context.tree_version, agent="coordinator")
    dd = await self.invoke_m1_decomposer_and_dependency_no_gate(
        context, on_agent_complete=on_agent_complete
    )
    merged.update(dd)
```

### 4.4 优化后完整流程

```
M2 一致性评估
    │
    ├── 规则引擎检查（5 项规则，0 token）
    │   └── 每条规则结果含 affected_nodes
    │
    ├── LLM 语义检查（1 次调用，新增 per_child_findings）
    │   └── 对每个节点输出 coverage_ok / faithfulness_ok / scope_ok
    │
    ├── _build_per_child_hints() 融合两类结果
    │   └── 输出 List[PerChildRefinementHint]
    │
    ▼
ConsistencyEvaluationResult（per_child 含真实数据）
    │
    ├── 通过 ──→ 可实现性 + 集成 ──→ 完成
    │
    └── 不通过
          │
          ├── per_child 有 need_further_split=True 的节点
          │       ↓
          │   祖先去重（父子都在失败集 → 只留父）
          │       ↓
          │   逐个调用 _refine_one_sub_ar_node 精准重拆
          │       ↓
          │   全量依赖重跑 + 重新生成子需求列表
          │       ↓
          │   回到一致性评估（最多 inner_max 轮）
          │
          ├── failed_node_ids 有具体节点（兜底一）
          │       ↓
          │   向上归约 → 逐个精准重拆
          │
          └── 无具体节点（兜底二）
                  ↓
              全树重拆（保留原逻辑，100% 向后兼容）
```

### 5.2 当前问题详解

当前 M2 可实现性评估共 6 类规则：

| 规则 | rule_id | severity | 问题 |
|------|---------|----------|------|
| 内聚度 | feasibility_001 | warning | 低内聚 warning 即触发重拆 |
| 功能点规模 | feasibility_002 | warning | 阈值 > 15 偏严 |
| 工作量估算 | feasibility_003 | warning | 阈值 > 0.5 人月偏严 |
| 粒度合理性 | feasibility_004 | warning | 过细/过粗都触发，且与 feasibility_002 重复 |
| 资源约束 | feasibility_005 | **error** | 唯一 error 级规则 |
| 技术复杂性 | feasibility_006 | warning | EO/ILF 单独命中即判失败，过于敏感 |

核心问题在于 `agent_invoker.py` 的 `_extract_feasibility_failed_node_ids`：

```text
遍历 critical_issues → 收集 affected_nodes
遍历 rule_results → 只跳过 passed=True → 收集其余所有 affected_nodes
```

即：**不区分 severity（error/warning）、不区分 issue_type、不区分粒度过细还是过粗**，全部触发自动细化。这就是"拆得很多"的核心原因。

### 5.3 方案 A：过滤自动细化触发条件（最直接）

修改 `_extract_feasibility_failed_node_ids`，只收集真正需要自动细化的规则：

```python
# 需要触发自动拆分的 issue_type
SPLIT_TRIGGER_ISSUE_TYPES = {
    "resource_constraint_mismatch",   # error 级
    "function_point_scale_issue",     # 规模过大（需验证 evidence）
    "granularity_issue",              # 只取 too_coarse，排除 too_fine
}

# 只需入报告、不触发自动拆分的 issue_type
NON_SPLIT_WARNING_ISSUE_TYPES = {
    "high_technical_complexity",
    "workload_exceeded",
    "low_cohesion",
}
```

收集节点时的过滤逻辑：

```python
def _extract_feasibility_failed_node_ids(eval_dict, ...) -> List[str]:
    failed_ids = set()

    for item in (eval_dict.get("rule_results") or []):
        if item.get("passed") is True:
            continue

        severity = item.get("severity", "warning")
        issue_type = item.get("issue_type", "")

        # warning 级且不在触发白名单 → 只入报告，不收集节点
        if severity == "warning" and issue_type in NON_SPLIT_WARNING_ISSUE_TYPES:
            continue

        # granularity_issue 只收集 too_coarse，不收集 too_fine
        if issue_type == "granularity_issue":
            evidence = item.get("evidence") or {}
            too_coarse = evidence.get("too_coarse_functions") or []
            for nid in too_coarse:
                failed_ids.add(str(nid).strip())
            continue  # 跳过通用 affected_nodes 收集

        for nid in item.get("affected_nodes") or []:
            s = str(nid).strip()
            if s:
                failed_ids.add(s)

    # 向上归约（复用维度二的归约逻辑）
    return _reduce_to_top_level_ancestors(failed_ids, current_function_list)
```

## 六、兼容性设计总表

| 场景 | 行为 |
|------|------|
| LLM 不返回 `per_child_findings` | `_build_per_child_hints` 仅用规则引擎数据 |
| `per_child` 为空列表 | 走 affected_nodes 分支，再不行回退全树重拆 |
| 旧版 LLM 模型不输出新字段 | `per_child_findings` 有默认值 `[]`，不影响流程 |
| `affected_nodes` 也为空 | 回退全树重拆（100% 保留原逻辑） |
| 问题节点相互为祖先-后代 | 向上归约自动去重，只保留最顶层祖先 |
| 精准重拆节点不在树中 | `_refine_one_sub_ar_node` 返回 False，记录 warning 跳过 |
| 可实现性 warning（技术复杂性/工作量） | 只入报告展示，不触发自动细化 |
| 可实现性粒度过细 | 不进入拆分流程，避免无限下钻 |
| 可实现性评估 | 复用相同的 per_child / affected_nodes 基础设施 |
| 可实现性 error（资源约束） | 继续触发自动细化，行为不变 |

## 七、预估效果

### 7.1 Token 消耗对比

假设 3 层功能树、20 个节点，仅 3 个节点一致性未通过：

| 阶段 | 改造前 | 改造后 |
|------|--------|--------|
| 一致性评估 LLM | 1 次（扁平 20 节点） | 1 次（树结构 + per_child 字段，~1.1× Token） |
| 重拆 Decomposer | 全量（20 节点上下文） | 3 个节点 × 子树上下文（~5 节点/个） |
| 重拆 Dependency | 全量（20 节点上下文） | 3 个节点 × 子树上下文（~5 节点/个） |
| 再评估一致性 | 全量（20 节点） | 全量（20 节点，但问题已修复） |
| **Token 总计估算** | **基准 100%** | **约 30–40%** |

### 7.2 质量提升汇总

| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| **一致性评估粒度** | 全量扁平评估 | 逐节点层级评估 |
| **per_child 数据** | 始终为空列表 | 含真实逐节点问题与建议 |
| **一致性重拆粒度** | 整棵功能树推倒 | 仅失败节点（向上归约后） |
| **通过节点稳定性** | 被无差别重拆，可能变差 | 保持不动，质量不退化 |
| **可实现性触发条件** | warning 也触发自动拆分 | warning 只入报告，不驱动拆分 |
| **可实现性阈值** | AFP > 15 / 工作量 > 0.5 | AFP > 25 / 工作量 > 1.0 |
| **技术复杂性判定** | EO/ILF 单独命中即失败 | 仅 HIGH + ILF/EIF 组合才警告 |
| **粒度过细** | 也进入拆分流程 | 不触发拆分 |
| **前端展示** | 无逐节点问题数据 | 可按节点展示具体问题及建议 |
| **兜底机制** | 无 | 双层兜底（affected_nodes → 全树重拆） |
| **向后兼容** | — | 所有旧路径保留，可 fallback 到原逻辑 |

### 7.3 风险与取舍

- 维度一/二：评估 LLM 的单次 Token 略增（~10%），但大幅减少了重拆 Token，总体净节省
- 维度三：放宽规则后树会更稳定，部分复杂节点的风险仅在报告中提示。用户可通过 `refine-node` 手动选择节点继续拆分
- 最终推荐策略：**自动细化只处理 error 或明确"规模严重过大"的节点；其余可实现性问题只作为报告建议，不驱动自动拆分**
