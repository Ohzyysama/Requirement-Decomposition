# 需求拆分评估与重拆优化方案

---

## 一、优化背景与问题诊断

### 1.1 当前流程概览

```
用户需求
    ↓
M1 Normalizer（需求标准化）
    ↓
M1 FunctionalDecomposer（功能拆分 → 整棵功能树）
    ↓
M1 DependencyClassifier（依赖分析）
    ↓
生成子需求列表（sub_requirement_list）
    ↓
M2 一致性评估（ConsistencyEvaluator）
    │
    ├── 通过 → M2 可实现性评估（FeasibilityEvaluator）
    │              │
    │              ├── 通过 → EvaluationIntegrator → 完成
    │              │
    │              └── 不通过 → 子 AR 递归细化（往往过度拆分）★ 核心问题
    │
    └── 不通过 → 内层重试（Decomposer+Dependency 重跑，默认最多 1 次）
                   ↓
               重新一致性评估 → 循环（或跳出继续）
```

### 1.2 核心问题：可实现性评估阈值过严导致过度拆分

| 编号 | 问题 | 根因 | 影响 |
|------|------|------|------|
| **问题一** | 可实现性评估**阈值过严**，大部分子功能都被判定为"不通过" | AFP > 15 即判大型功能、工作量 > 0.5 人月即判过高、EO/ILF 类型直接判复杂 | 几乎每个 AR 都会触发细化，拆分层数失控 |
| **问题二** | **warning 级规则也触发自动重拆**，不区分严重程度 | `_extract_feasibility_failed_node_ids` 收集所有 `passed=False` 的 `affected_nodes`，不区分 severity | 技术复杂性、工作量偏高、低内聚等次要问题也导致拆分，树无限下钻 |
| **问题三** | FPA 功能点计算**纯规则驱动、不利用 LLM 语义理解** | `_determine_function_type` 基于关键词匹配，`_determine_complexity` 基于描述长度 | 功能分类不准、复杂度估得偏高，进一步加剧阈值过严问题 |
| **问题四** | **阈值缺乏开发者经验维度**，未利用 LLM 估算实际开发时间 | 当前只按 FPA 公式计算（6.72 人时/功能点 × AFP ÷ 174），不与开发者能力匹配 | 大型项目中的正常功能也被判定为"过大"，忽略上下文与实际开发能力 |

### 1.3 问题根源详解

#### 1.3.1 当前可实现性阈值一览

| 规则 | rule_id | severity | 当前阈值 | 问题 |
|------|---------|----------|----------|------|
| 功能点规模 | feasibility_002 | warning | AFP > 15 | 一个中等 ILF（数据库表 CRUD）= 10~15 FP，再加上关联查询就直接超标 |
| 工作量估算 | feasibility_003 | warning | > 0.5 人月 | 0.5 人月 ≈ 10 个工作日，对于中等复杂度的功能非常常见，不应触发拆分 |
| 粒度合理性 | feasibility_004 | warning | too_coarse: AFP > 15; too_fine: AFP < 1 | 与 feasibility_002 重复；粒度过细也收集 affected_nodes，理论上触发"合并"而非"拆分"但处理逻辑未区分 |
| 资源约束 | feasibility_005 | **error** | 工作量 > available_workload_per_function（默认 1.0 人月） | 唯一 error 级，但 available_workload 在 context 中几乎不会配置，所以这规则通常不会命中 |
| 技术复杂性 | feasibility_006 | warning | complexity == "HIGH" 或 function_type in ["EO", "ILF"] | "生成报表"（EO）、"存储用户数据"（ILF）等常规功能直接判复杂——这是最不合理的规则 |
| 内聚度 | feasibility_001 | warning | LLM 判断 low_cohesion | LLM 判断本身合理，但不应直接触发自动拆分 |

#### 1.3.2 `_extract_feasibility_failed_node_ids` 的问题

[agent_invoker.py](backend_release/backend/app/services/coordinator/agent_invoker.py) 第 206-226 行：

```python
@staticmethod
def _extract_feasibility_failed_node_ids(feas: Dict[str, Any]) -> List[str]:
    out: List[str] = []
    seen: set = set()

    def _collect_from_rule(item: Any) -> None:
        if not isinstance(item, dict):
            return
        if item.get("passed") is True:
            return
        # ─── 问题：不区分 severity(error/warning)、不区分 issue_type ───
        for n in item.get("affected_nodes") or []:
            s = str(n).strip()
            if s and s not in seen:
                seen.add(s)
                out.append(s)

    for item in feas.get("critical_issues") or []:
        _collect_from_rule(item)
    for item in feas.get("rule_results") or []:
        _collect_from_rule(item)
    return out
```

即：**warning 级的技术复杂性、工作量偏高、低内聚、粒度过粗——全部触发 `_feasibility_failed_walk` 递归细化**。这就是"拆得很多"的根本原因。

#### 1.3.3 FPA 功能分类不利用 LLM

[m2_feasibility_agent.py](backend_release/backend/app/services/agents/M2/agents/m2_feasibility_agent.py) 第 231-261 行：

```python
def _determine_function_type(self, func_desc: str) -> str:
    """确定功能类型——纯关键词匹配，不利用 LLM 语义理解"""
    # "添加用户" → EI（权重 3-6）
    # "显示报表" → EO（权重 4-7）
    # "查询列表" → EQ（权重 3-6）
    # "存储数据" → ILF（权重 7-15）← 权重最高，极易超标
    # "对接API" → EIF（权重 5-10）
```

`_determine_complexity` 按描述文本长度分级：< 20 字为 LOW，< 50 字为 MEDIUM，>= 50 字为 HIGH。这导致描述详细的正常功能被误判为 HIGH 复杂度，进一步推高 AFP。

## 二、优化思路总览

优化分为三个互补维度：

```
┌──────────────────────────────────────────────────────────────────┐
│  维度一：可实现性触发条件分级（过滤侧）── 最直接控制"拆得多"         │
│  · error 级 → 触发自动细化（当前仅 feasibility_005 资源约束）       │
│  · warning 级 → 只入报告、不触发自动拆分                           │
│  · 粒度过细 → 不进入拆分流程                                      │
│  · 技术复杂性 EO/ILF → 不再自动判失败                             │
└───────────────────────────┬──────────────────────────────────────┘
                            │
                            ▼ 控制哪些规则真正触发重拆
┌──────────────────────────────────────────────────────────────────┐
│  维度二：可实现性阈值合理化 + AI 参与估算（规则侧）                  │
│  · LLM 估算开发者实现时间（替代纯 FPA 公式阈值）                    │
│  · 放宽 AFP 阈值（15→按开发者经验动态调整）                        │
│  · 放宽工作量阈值（0.5→1.0 人月，或由 LLM 判断是否合理）            │
│  · FPA 功能分类引入 LLM 语义判断（替代纯关键词匹配）                 │
│  · 修正 max_feasibility_refinement_depth Schema 文案（3→1）        │
└───────────────────────────┬──────────────────────────────────────┘
                            │
                            ▼ 让评估结果更贴近真实开发场景
┌──────────────────────────────────────────────────────────────────┐
│  维度三：子 AR 拆分深度控制（深度侧）── 已实现，需收尾               │
│  · max_feasibility_refinement_depth 默认 1（已实现）               │
│  · 即以当前拆分根为基准，只对一层子节点做细化                       │
│  · 修正 coordinator.py 中错误提示文案（"默认值 3" → "默认值 1"）    │
└──────────────────────────────────────────────────────────────────┘
```

### 核心原则

> **过滤侧**：只有明确"规模严重过大"（error 级）才触发自动拆分；warning 级问题只在报告中展示，由用户自主决定是否手动 refine-node
>
> **规则侧**：引入 LLM 估算实际开发时间（"X 年经验的开发者实现此功能需要 Y 天"），替代纯公式 AFP 阈值；FPA 分类由 LLM 语义判断而非关键词匹配
>
> **深度侧**：默认只拆分一层（已实现），用户可通过配置调整层数或手动对指定节点 refine-node

## 三、维度一：可实现性触发条件分级（过滤侧）

### 3.1 改动文件

| 文件 | 改动类型 | 改动量 |
|------|----------|--------|
| [agent_invoker.py](backend_release/backend/app/services/coordinator/agent_invoker.py) | 修改 `_extract_feasibility_failed_node_ids` | ~50 行 |

### 3.2 分级策略

```
触发自动细化的条件（SPLIT_TRIGGER）:
  ✅ severity == "error"                        → 资源约束严重不匹配
  ✅ issue_type == "function_point_scale_issue" → AFP 严重超标（> 阈值上限 × 2）
  ✅ issue_type == "granularity_issue"          → 仅 too_coarse（排除 too_fine）

只入报告、不触发自动细化的条件（REPORT_ONLY）:
  ❌ severity == "warning" + issue_type == "low_cohesion"
  ❌ severity == "warning" + issue_type == "workload_exceeded"
  ❌ severity == "warning" + issue_type == "high_technical_complexity"
  ❌ severity == "warning" + issue_type == "granularity_issue" → too_fine 子集
```

### 3.3 代码改动

```python
# ─── 触发白名单：只有这些 issue_type 才进入自动细化 ───
_SPLIT_TRIGGER_ISSUE_TYPES: frozenset[str] = frozenset({
    "resource_constraint_mismatch",    # error 级：资源约束不匹配
    "function_point_scale_issue",      # AFP 严重超标（需额外验证 evidence）
})

# ─── 显式排除：这些 issue_type 只入报告不触发拆分 ───
_REPORT_ONLY_ISSUE_TYPES: frozenset[str] = frozenset({
    "low_cohesion",                    # 低内聚：应由人工判断是否重拆
    "workload_exceeded",               # 工作量偏高：大型项目正常现象
    "high_technical_complexity",       # 技术复杂：需人工评估，不应自动拆
})


@staticmethod
def _extract_feasibility_failed_node_ids(
    feas: Dict[str, Any],
    function_list: Optional[List[Dict[str, Any]]] = None,
) -> List[str]:
    """从可实现性评估 dict 收集需要自动细化的节点 id。

    分级规则：
      - severity == "error"                         → 收集
      - severity == "warning" + 触发白名单           → 收集
      - severity == "warning" + REPORT_ONLY          → 跳过（只入报告）
      - granularity_issue                           → 只收集 too_coarse，排除 too_fine
    """
    out: List[str] = []
    seen: set = set()

    def _add(nid: str) -> None:
        s = str(nid).strip()
        if s and s not in seen:
            seen.add(s)
            out.append(s)

    for item in (feas.get("rule_results") or []):
        if not isinstance(item, dict):
            continue
        if item.get("passed") is True:
            continue

        severity = str(item.get("severity", "")).lower()
        issue_type = str(item.get("issue_type", ""))

        # ── error 级一律收集 ──
        if severity == "error":
            for n in item.get("affected_nodes") or []:
                _add(str(n).strip())
            continue

        # ── warning 级：只收集白名单中的类型 ──
        if issue_type in _REPORT_ONLY_ISSUE_TYPES:
            continue

        # ── granularity_issue：只收集 too_coarse ──
        if issue_type == "granularity_issue":
            evidence = item.get("evidence") or {}
            too_coarse = evidence.get("too_coarse_functions") or []
            for nid in too_coarse:
                _add(str(nid).strip())
            continue

        # ── function_point_scale_issue：需额外验证 evidence 中的 AFP 是否确实严重超标 ──
        if issue_type == "function_point_scale_issue":
            evidence = item.get("evidence") or {}
            threshold = evidence.get("threshold", 25)
            # 仅当 AFP > 阈值 × 2 时才触发（即 AFP > 50）
            if threshold < 50:
                continue

        # ── 默认收集 ──
        for n in item.get("affected_nodes") or []:
            _add(str(n).strip())

    # 向上归约去重
    if function_list:
        out = _reduce_to_top_level_ancestors(out, function_list)
    return out
```

### 3.4 效果

改造后的行为：
- 技术复杂性高的功能 → **报告中展示**，不触发自动细化
- 工作量偏高 → **报告中展示**，不触发自动细化
- 低内聚 → **报告中展示**，不触发自动细化
- 功能点规模超过 25 但未超过 50 → **报告中展示**，不触发自动细化
- 功能点规模超过 50（阈值 × 2）→ 触发自动细化
- 资源约束 error → 触发自动细化
- 粒度过细 → 不触发拆分（在报告中建议合并）

### 4.2 核心思路：让 LLM 参与阈值判断

当前系统的 FPA 计算完全是规则驱动的：

```
func_desc → 关键词匹配 → function_type (EI/EO/EQ/ILF/EIF)
         → 描述长度  → complexity (LOW/MEDIUM/HIGH)
         → 查权重表  → function_points
         → × TCF     → AFP
         → × 6.72    → 工作量（人时）
         → ÷ 174     → 工作量（人月）
```

问题在于：
1. **关键词匹配不准**："用户管理模块" 有哪些关键词？"输入"？"存储"？"查询"？可能同时命中多个
2. **描述长度 ≠ 复杂度**：一句 60 字的啰嗦描述不等于 HIGH 复杂度
3. **FPA 阈值不与开发者经验挂钩**：AFP=20 对初级开发可能是 1 人月，对高级开发可能只需 0.3 人月

**优化方向**：引入 LLM 对每个子功能做**语义级 FPA 分类 + 开发时间估算**，结果作为阈值判断的主要依据。

### 4.3 新增 LLM 估算：开发者经验 → 实现时间

在 `m2_feasibility_agent.py` 的 `_perform_per_function_fpa_analysis` 中，新增一个 LLM 调用来估算每个功能的开发时间：

```python
class DeveloperEffortEstimate(BaseModel):
    """LLM 对单个子功能的开发工作量估算"""
    function_id: str = Field(..., description="功能节点 ID")
    
    # ── FPA 语义分类（替代关键词匹配）──
    function_type: str = Field(
        ..., description="功能类型: EI/EO/EQ/ILF/EIF",
        pattern="^(EI|EO|EQ|ILF|EIF)$"
    )
    complexity: str = Field(
        ..., description="复杂度: LOW/MEDIUM/HIGH",
        pattern="^(LOW|MEDIUM|HIGH)$"
    )
    classification_reason: str = Field(
        default="", description="分类理由"
    )
    
    # ── 按开发者经验估算实现时间（核心新增）──
    junior_dev_days: float = Field(
        ..., description="1-2年经验的初级开发者预计实现天数（含自测）"
    )
    mid_dev_days: float = Field(
        ..., description="3-5年经验的中级开发者预计实现天数（含自测）"
    )
    senior_dev_days: float = Field(
        ..., description="5年+经验的高级开发者预计实现天数（含自测）"
    )
    
    # ── 拆分合理性判断 ──
    needs_further_split: bool = Field(
        default=False,
        description="基于经验判断，该功能是否建议进一步拆分"
    )
    split_reason: Optional[str] = Field(
        default=None, description="如需拆分，说明原因"
    )
    
    # ── 置信度 ──
    confidence: float = Field(default=0.8, ge=0.0, le=1.0)


class PerFunctionEffortEstimateResult(BaseModel):
    """批量功能开发工作量估算结果"""
    estimates: List[DeveloperEffortEstimate] = Field(
        default_factory=list
    )
    overall_notes: Optional[str] = Field(default=None, description="整体说明")
```

### 4.4 阈值判断逻辑改造

利用 LLM 估算结果替换纯 FPA 公式阈值：

```python
# ─── 旧逻辑（纯公式）───
# if function_afp > 15:  → 判定为过大
# if function_workload > 0.5: → 判定为工作量过高

# ─── 新逻辑（LLM 估算 + 公式兜底）───
def _should_trigger_refinement(
    llm_estimate: DeveloperEffortEstimate,
    fpa_afp: float,
    fpa_workload: float,
) -> Tuple[bool, Optional[str]]:
    """
    综合 LLM 估算和 FPA 公式判断是否需要触发细化。
    
    判断优先级：
      1. LLM 明确建议拆分（needs_further_split=True）
         → 触发，附带 LLM 的理由
      2. LLM 估算中级开发者 > 15 天 → 触发
      3. FPA 公式兜底：AFP > 50 → 触发（仅当 LLM 估算不可用时）
      4. 其他情况 → 不触发
    """
    # 优先级 1：LLM 明确建议
    if llm_estimate.needs_further_split:
        return True, llm_estimate.split_reason or "LLM 评估建议进一步拆分"
    
    # 优先级 2：按中级开发者估算（取最贴近团队实际的参考值）
    if llm_estimate.mid_dev_days > 15:
        return True, (
            f"中级开发者预计需 {llm_estimate.mid_dev_days:.0f} 天，"
            f"超过单功能合理上限 15 天"
        )
    
    # 优先级 3：FPA 公式兜底（阈值放宽到 50）
    if fpa_afp > 50:
        return True, f"FPA 功能点 {fpa_afp:.1f} > 50，规模过大"
    
    # 不触发
    return False, None
```

### 4.5 LLM Prompt 设计

关键 Prompt 要点：

```
你是一个有 10 年经验的软件架构师和全栈开发者。请对以下子功能列表中的
每一个功能，完成两件事：

1. 【FPA 分类】根据 IFPUG 标准判断功能类型（EI/EO/EQ/ILF/EIF）和复杂度
   （LOW/MEDIUM/HIGH），不要仅凭关键词，要理解功能描述的语义。

2. 【开发时间估算】假设是一个典型的中型互联网项目（前后端分离、有数据库、
   有基本的 CI/CD），分别估算三种经验水平的开发者实现该功能需要的**开发天数**：
   - 初级：1-2 年经验，需要较多指导和调试时间
   - 中级：3-5 年经验，能独立完成但不一定是最优方案
   - 高级：5 年+经验，能快速给出高质量实现
   
   估算时请考虑：
   - 该功能的 CRUD 复杂度、业务逻辑复杂度
   - 前后端工作量的比例
   - 是否需要对接第三方服务
   - 测试和联调时间（含 20% buffer）
   - 不要考虑需求文档编写、code review、部署等非开发活动

3. 【拆分建议】基于中级开发者的估算，如果超过 15 天，请判断是否建议
   进一步拆分，并说明拆分的具体方向。
```

### 4.6 放宽 FPA 规则阈值（作为兜底）

当 LLM 估算不可用或结果置信度低时，回退到放宽后的 FPA 规则：

```python
# _check_function_point_scale_per_function
# - 旧阈值: function_afp > 15
# + 新阈值: function_afp > 25
# + 严重超标: function_afp > 50（触发自动细化，见维度一）

# _check_workload_estimation_per_function  
# - 旧阈值: function_workload > 0.5    # 0.5 人月
# + 新阈值: function_workload > 1.0    # 1.0 人月

# _check_granularity_reasonableness_per_function
# - 旧: too_coarse: function_afp > 15
# + 新: too_coarse: function_afp > 25
# - too_fine: function_afp < 1（不触发拆分，只入报告）

# _check_technical_complexity_per_function
# - 旧: complexity == "HIGH" or function_type in ["EO", "ILF"]
# + 新: complexity == "HIGH" and function_type in ["ILF", "EIF"]
```

### 5.3 深度语义说明

`max_feasibility_refinement_depth = n` 的含义：

```
当前拆分根（F-1 或用户选择的 refine-node 节点）
    ├── 第 0 层（根本身，深度 = 0）
    │
    ├── 第 1 层子节点 ← 深度 = 1，max_feasibility_refinement_depth=1 时可细化这一层
    │   ├── 第 2 层子节点 ← 深度 = 2，max_feasibility_refinement_depth=2 时可细化到这一层
    │   │   ├── ...
```

对于正常拆分（根为 F-1）：
- `max_feasibility_refinement_depth = 1`（默认）：只对 F-1 的直接子节点（第 1 层）做可实现性细化
- `max_feasibility_refinement_depth = 2`：可细化到孙节点
- `max_feasibility_refinement_depth = 0`：完全禁止自动细化（需配合 `enable_feasibility_refinement=false`）

对于 `refine-node`（根为用户选择的节点）：
- 以用户选择的节点为第 0 层，同样语义
- 可通过 `RefineNodeRequest.config.max_feasibility_refinement_depth` 单独设置

## 八、预估效果

### 8.1 拆分深度控制效果

假设 3 层功能树、15 个叶子节点：

| 指标 | 改造前 | 改造后 |
|------|--------|--------|
| 自动细化触发节点数 | ~12-15 个（几乎所有 warning 都触发） | ~2-4 个（仅 error + 严重超标） |
| 单次任务 LLM 调用次数 | 基础 8 次 + 细化 N×4 次 | 基础 9 次（+1 次 LLM 估算）+ 细化~3×4 次 |
| 最大树深度 | 可能无限下钻 | 受 max_feasibility_refinement_depth 控制（默认 1） |
| 用户可见的 warning 信息 | 少（部分被自动拆分淹没） | 完整（所有 warning 都在报告中有记录） |

### 8.2 质量影响

| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| **FPA 功能分类** | 纯关键词匹配，准确率 ~60% | LLM 语义判断 + 关键词兜底，准确率 ~90% |
| **复杂度判定** | 描述长度 → 复杂度（粗糙） | LLM 语义理解 + 描述长度兜底 |
| **开发时间估算** | FPA 公式（脱离实际） | LLM 按经验水平估算 + FPA 公式兜底 |
| **触发拆分条件** | warning 即触发 | error + 严重超标才触发 |
| **粒度过细处理** | 也进入拆分流程 | 不入拆分，报告中建议合并 |
| **技术复杂性** | EO/ILF 自动判复杂 | 仅 HIGH+ILF/EIF 才警告 |
| **默认拆分深度** | Schema 写 3 实际 1（不一致） | 统一为 1 |
| **用户可控性** | 自动拆分不可控 | warning 在报告中可见，用户手动 refine-node |

### 8.3 风险与取舍

- **维度一（过滤侧）**：部分复杂功能不再自动细化，风险转移到报告中。用户可通过 `refine-node` 手动选择节点拆分 → 建议在前端评估报告面板中突出展示 warning 信息，引导用户关注
- **维度二（规则侧）**：新增 1 次 LLM 调用，但换来的是一次调用即可获得 FPA 分类 + 复杂度 + 开发时间估算，净收益为正（减少无效细化调用）；LLM 估算结果不可靠时，放宽后的 FPA 公式作为兜底，不降低功能可用性
- **维度三（深度侧）**：默认为 1 层已在代码中实现，仅修正 Schema 文案 → 零风险
